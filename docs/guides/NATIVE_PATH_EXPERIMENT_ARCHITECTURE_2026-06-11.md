# Native Path Experiment Architecture

Date: 2026-06-11

This document reorganizes the current project direction after the decision to restrict explainability scoring to models with native recommendation paths.

## 1. Current Research Position

The current paper direction should not be framed as "evaluate explainability for every KG recommender."

The stricter and more defensible framing is:

```text
Evaluate the accuracy-explainability tradeoff of KG recommenders whose explanations are native to the recommendation mechanism, while using strong non-path KG recommenders only as accuracy references.
```

This matters because post-hoc paths do not faithfully represent the model's actual decision process. Therefore:

- native path models can be evaluated with `NDCG / HR / Recall / Precision` and `LIR / SEP / ETD`,
- non-path models can be evaluated with recommendation accuracy only,
- post-hoc path recovery should not be used to compute the main explainability scores.

## 2. Experiment Groups

### Main Group: Native Path Models

These models are the core experimental targets. They must provide paths that are part of, or directly generated by, the recommendation process.

Current targets:

- `PGPR`
- `UCPR`
- `CAFE`

Possible additions:

- other KG recommendation models with native path generation,
- only if their path output is faithful to the recommendation mechanism.

Metrics:

```text
NDCG
HR
Recall
Precision
LIR
SEP
ETD
```

### Reference Group: Strong Non-Path KG Recommenders

These models are useful as accuracy references, but they should not enter the main explainability-score comparison unless they provide native recommendation paths.

Possible targets:

- `KGIN`
- `KGAT`
- `MKR`
- `CKE`
- `RippleNet`
- `LightGCN` or KG-aware variants

Metrics:

```text
NDCG
HR
Recall
Precision
```

Excluded from these models:

```text
LIR
SEP
ETD
```

Reason:

```text
No native path object means no faithful path explainability score.
```

## 4. Required Output Contract

All native path models must eventually export the following canonical-format files:

```text
uid_topk.csv
pred_paths.csv
uid_pid_explanation.csv
```

All ids in these files must be canonical `uid/pid`.

Non-path models only need:

```text
uid_topk.csv
```

The evaluation layer should reject or skip explainability scoring when native path files are absent or not faithful.

## 9. Expected Analysis Logic

Possible claims if supported by data:

1. Native path models provide faithful path explanations but may trail stronger latent/GNN models in recommendation accuracy.
2. Different native path mechanisms produce different tradeoffs across `LIR`, `SEP`, `ETD`, and accuracy.
3. Non-path KG recommenders should be treated as accuracy references, not explainability baselines.
4. Canonical dataset alignment is necessary because otherwise models may be compared across different user/item/split/label spaces.
5. xrecsys remains useful as a loose-coupled evaluation layer, but only when the model output satisfies the native-path contract.
